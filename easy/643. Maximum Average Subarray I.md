# [643. Maximum Average Subarray I](https://leetcode.com/problems/maximum-average-subarray-i/description/?envType=study-plan-v2&envId=leetcode-75)

## Intuition

Since the problem deals with a contiguous subarray,<br>
we can apply the **sliding window** technique to reuse previous computations efficiently.

---

## Approach
1. First, calculate the sum of the initial window of size `k`.
2. Move the window one step at a time:
   - Subtract the element that is leaving the window
   - Add the new element entering the window
   - → This allows updating the sum in `O(1)` time.
3. Update the maximum sum at each step.
4. After completing the traversal, return the maximum average by dividing the maximum sum by `k`.

---

## Complexity
### Time Complexity
O(n), where n is the length of the array

### Space Complexity
O(1), as only a few variables are used regardless of input size

---

## My Solution
```
class Solution {
    public double findMaxAverage(int[] nums, int k) {
        double answer = 0;

        for (int i = 0; i < k; i++) {
            answer += nums[i];
        }

        double maxValue = answer;
        for (int i = k; i < nums.length; i++) {
            answer = answer + nums[i] - nums[i - k];
            maxValue = Math.max(maxValue, answer);
        }

        return maxValue / k;
    }
}
```

---

## Review
The original solution already achieves optimal time and space complexity,<br>
`O(n)` and `O(1)`, by **using the sliding window technique.**

Therefore, there is no room for algorithmic improvement, and any optimization is limited to constant factors.

- `double` → `int` accumulation:
  - Integer operations are generally faster than floating-point operations, and division can be deferred until the end.
- `Math.max()` → explicit `if` comparison:
  - Although `Math.max()` is typically inlined by the JIT compiler, an explicit comparison can be more straightforward.
- Renamed variables to `windowSum` and `maxSum` for better readability and clarity.

---

## Revised Solution
```
class Solution {
    public double findMaxAverage(int[] nums, int k) {
        int windowSum = 0;

        for (int i = 0; i < k; i++) {
            windowSum += nums[i];
        }

        int maxSum = windowSum;
        for (int i = k; i < nums.length; i++) {
            windowSum += nums[i] - nums[i - k];
            if (windowSum > maxSum) maxSum = windowSum;
        }

        return (double) maxSum / k;
    }
}
```

### Comparison Summary

| Category         | Before               | After                |
| ---------------- |----------------------|----------------------|
| Max calculation  | Uses `Math.max`      | Uses `if` comparison |
| Variable naming  | `answer` (ambiguous) | `windowSum` (clear)  |
| Time Complexity  | O(n)                 | O(n)                 |
| Space Complexity | O(1)                 | O(1)                 |

This improvement is not a change in the algorithm itself,<br>

### Key Improvements
- **Accumulator Type Optimization**
  - Changed accumulation variable from `double` to `int`
  - Deferred the only floating-point division to the final return statement 
- **Runtime Efficiency**
  - Runtime dropped from 5ms to 3ms,
  - and the percentile ranking improved significantly from **39.47% to 81.37%.**
  - → Replacing `double` accumulation with `int` eliminates per-iteration floating-point overhead, as integer arithmetic is natively faster on the JVM. 
- **Memory Efficiency**
  - Absolute memory usage dropped from 69.69MB to 69.20MB,
  - and the percentile ranking improved significantly from **43.68% to 93.60%.**
  - → Using a primitive `int` instead of `double` reduces the JVM's internal bookkeeping and GC pressure during execution.
- **Max Comparison Simplification**
  - Before: `Math.max(maxValue, answer)` — method call with boxing overhead risk.
  - After: `if (windowSum > maxSum) maxSum = windowSum` — direct inline branch.
  - → Removes unnecessary method dispatch and improves branch predictor friendliness.
