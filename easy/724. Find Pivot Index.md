# [724. Find Pivot Index](https://leetcode.com/problems/find-pivot-index/description/?envType=study-plan-v2&envId=leetcode-75)

## Intuition

The sum of elements to the left and right of the pivot index must be equal.<br>
The most intuitive approach is a **brute-force** search that directly computes both sides for every index.
---

## Approach

1. Iterate through all indices, treating each one as a potential pivot.
2. For each pivot, compute the sum of the left subarray and the sum of the right subarray separately.
3. If both sums are equal, return that index.
4. If no valid pivot is found by the end, return -1.

---

## Complexity
### Time Complexity
O(n²), since for each index we recompute the left and right sums using nested loops.

### Space Complexity
O(1), as only a few variables are used regardless of input size

---

## My Solution
```
class Solution {
    public int pivotIndex(int[] nums) {

        for (int pivotIndex = 0; pivotIndex < nums.length; pivotIndex++) {
            int leftSum = 0, rightSum = 0;

            for (int j = 0; j < pivotIndex; j++) {
                leftSum += nums[j];
            }

            for (int k = pivotIndex + 1; k < nums.length; k++) {
                rightSum += nums[k];
            }

            if (leftSum == rightSum) return pivotIndex;
        }

        return -1;
    }
}
```

---

## Review
The current approach is **inefficient** because it recalculates the left and right sums for every pivot.<br>
We can eliminate this redundancy and significantly improve performance.

1. Compute the total sum (`totalSum`) upfront.
2. Accumulate only the left sum (`leftSum`) as we iterate.
3. Compute the right sum implicitly using: `rightSum = totalSum - leftSum - nums[i]`

This reduces the problem to a single pass, bringing the time complexity down to O(n).

---

## Revised Solution
```
class Solution {
    public int pivotIndex(int[] nums) {
        int totalSum = 0;
        for (int num : nums) totalSum += num;

        int leftSum = 0;
        for (int i = 0; i < nums.length; i++) {
            if (leftSum == totalSum - leftSum - nums[i]) return i;
            leftSum += nums[i];
        }
        return -1;
    }
}
```

### Comparison Summary

| | Before | After |
|---|---|---|
| Time Complexity | O(n²) | **O(n)** |
| Space Complexity | O(1) | O(1) |
| Iterations | Up to n² | **Up to 2n** |

---

## Performance Comparison

This improvement is not a change in the algorithm itself,<br>
but rather an **implementation refinement focused on eliminating redundant recomputation of prefix sums.**

### Key Improvements
- **Algorithm Complexity Optimization**
  - Changed from a nested loop (O(n²)) to a two-pass linear scan (O(n))
  - Precomputed the total sum once, then derived rightSum on-the-fly during pivot traversal
- **Runtime Efficiency**
  - Runtime dropped from 323ms to 1ms,
  - and the percentile ranking improved significantly from **5.37% to 98.63%.**
  - → Eliminating the inner loops removes redundant summation on every pivot iteration, as each element is now visited at most twice total.
- **Memory Efficiency**
  - Absolute memory usage dropped from 47.50MB to 47.03MB,
  - and the percentile ranking improved significantly from **39.80% to 92.57%.**
  - → Replacing two auxiliary loop variables per iteration with a single running `leftSum` accumulator reduces stack frame overhead during execution.
- **rightSum Computation Simplification**
  - Before: explicit inner loop iterating from `pivot+1` to end — O(n) per pivot.
  - After: `totalSum - leftSum - nums[i]` — single arithmetic expression.
  - → Removes the inner loop entirely and replaces it with constant-time arithmetic, dramatically improving cache locality and branch predictor behavior.
